| 更新说明 | 新增预加载功能（FR-25 ~ FR-29），优化滑动流畅度与用户体验 |

---

## 一、需求背景

用户需要通过**滑动卡片**的方式快速浏览随机图片，并对感兴趣的图片进行**保存**操作，不感兴趣的图片直接**跳过**。该交互方式借鉴Tinder式左右滑动，操作直观、效率高。

---

## 二、功能目标

- 用户进入页面后自动加载一张随机图片
- 通过左右滑动完成「保存」与「跳过」操作
- 支持「撤销」上一次操作
- 交互流畅，反馈清晰
- **通过预加载机制消除滑动后的等待时间，实现零延迟滑动体验**

---

| 角色 | 描述 |
|------|------|
| 普通用户 | 浏览图片、保存图片、跳过图片 |

| 需求编号 | 需求描述 | 优先级 |
|---------|---------|--------|
| FR-22 | 当接口返回**空数据**或**无更多图片**时，展示空状态页面（文案：「暂无更多图片」+ 刷新按钮） | P1 |
| FR-23 | 网络异常时展示网络错误提示，并提供**重试**按钮 | P1 |
| FR-24 | **无重复图片**：接口保证每次返回的图片均不相同，客户端无需做去重处理 | P0 |

### 4.7 预加载功能（V1.1 新增）

| 需求编号 | 需求描述 | 优先级 |
|---------|---------|--------|
| FR-25 | 应用启动后，在加载首张图片的同时，**异步预加载下一张图片**并缓存（图片数据含URL、ID及Bitmap/Drawable） | P0 |
| FR-26 | 每次卡片滑出后（无论左滑还是右滑），**立即将缓存的下一张图片切换为当前展示卡片**，同时**异步预加载新的后续图片**，确保缓存池中始终至少有一张图片待用 | P0 |
| FR-27 | 预加载队列容量固定为 **1张**（即最多缓存1张未来图片），避免内存占用过高 | P1 |
| FR-28 | 当用户进行**撤销**操作时，预加载逻辑做如下处理：<br>- 撤销返回的图片直接展示，**不重新请求**（复用已存在数据）<br>- 撤销后**重置**预加载队列，重新预加载下一张图片，确保数据一致性 | P1 |
| FR-29 | 图片缓存的生命周期跟随当前Session，页面销毁时**释放所有预加载缓存**，避免内存泄漏 | P1 |

## 五、预加载架构设计

### 5.1 数据流示意
┌─────────────────────────────────────────────────────────────────────┐
│ 预加载数据流 │
├─────────────────────────────────────────────────────────────────────┤
│ │
│ ┌──────────┐ ┌──────────┐ ┌──────────┐ │
│ │ 当前卡片 │ ──▶ │ 卡片滑出 │ ──▶ │ 展示缓存 │ ──▶ 循环 │
│ └──────────┘ └──────────┘ └──────────┘ │
│ │ │ │
│ ▼ ▼ │
│ ┌──────────┐ ┌──────────┐ ┌──────────┐ │
│ │ 预加载 │ │ 后台 │ │ 缓存池 │ │
│ │ 触发 │ ──▶ │ 请求 │ ──▶ │ 填充 │ │
│ └──────────┘ └──────────┘ └──────────┘ │
│ │
│ 说明： │
│ 1. 首屏加载时：同时发起当前图片请求 + 预加载请求 │
│ 2. 滑动切换时：展示缓存 → 触发新的预加载请求 │
│ 3. 撤销操作时：展示撤销图片 → 清空缓存 → 重新预加载 │
│ 4. 最大缓存数：1张 │
└─────────────────────────────────────────────────────────────────────┘

### 5.2 预加载状态机

| 状态 | 描述 | 触发条件 |
|------|------|---------|
| IDLE | 无预加载任务 | 初始状态 / 所有任务完成 |
| LOADING | 正在请求下一张图片 | 触发预加载 |
| CACHED | 图片已缓存就绪 | 图片下载完成 |
| EMPTY | 无更多图片可加载 | 接口返回空数据 |
